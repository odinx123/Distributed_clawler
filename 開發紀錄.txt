使用 Redis:
1. 避免重複取件： Redis 的 LPOP 或 RPOP 操作是**原子性（Atomic）**的。這保證了同一個 URL 只會被一個 Worker 領走，不會發生兩個 Worker 同時抓取同一個網頁的情況。
2. 去重效率： 利用 Redis 的 Set 或 Bloom Filter 進行 URL 去重（Seen-set），可以在極短時間內判斷一個網址是否已經處理過，避免陷入死循環或浪費頻寬。
3. 如果隊列只存在於某個 Worker 的記憶體中，一旦程序當機，所有待抓取的任務都會消失。
4. 在傳統設計中，如果爬蟲程式（Worker）把「待抓清單」存在自己電腦的記憶體裡，這台電腦就是瓶頸。這時候如果想要開第二台電腦，第二台電腦不知道第一台抓了什麼，兩者無法溝通。
    - Redis 的解決方案： Redis 把「任務狀態」從爬蟲程式中抽離出來。
    - 無狀態化 (Stateless)： 爬蟲程式變得像是一台「純代工機器」，它不存資料，只管從 Redis 拿網址、抓資料、交件。
    - 隨插即用： 既然網址都在 Redis 這個「公共置物櫃」裡，你隨時可以多開 10 台、100 台電腦。每台電腦只要連上 Redis，就能立刻開始分擔工作，而不需要互相打招呼。
5. 如果任務清單只存在程式的變數（如 Python 的 list）裡，一旦當機或停電，記憶體裡的資料會立刻清空，會忘記還有哪些網址沒抓。
    - Redis 雖然是記憶體資料庫，但它有「自動存檔」機制：
        - 它會定時把記憶體裡的快照存到硬碟（RDB）。
        - 或者把每一次的操作紀錄寫成日誌（AOF）。
    - 任務確認機制 (Ack/Reliability):
        - 這是最核心的韌性設計。假設 Worker A 從隊列拿走了一個網址，但在解析網頁時突然當機了：
            - 一般隊列： 拿走就沒了，這個網頁永遠沒被抓到，形成資料遺漏。
            - Redis 模式： 我們可以使用 RPOPLPUSH 這種指令。它在把任務給 Worker 的同時，會先在 Redis 內部把這個任務備份到一個「處理中」的清單。
                - 如果 Worker 成功完成，就刪除備份。
                - 如果 Worker 斷線，系統發現備份還在，就會把任務丟回主隊列讓別人重做。
面試:
- 面試官需要的是**「分散式思維」**。在面試中，面試官想聽的不是你花了多少錢買硬體，而是你如何處理併發（Concurrency）、同步（Synchronization）與資源競爭（Race Condition）。
    - 重點在於： 你的代碼必須寫得像是「我完全不知道誰會來領任務」。如果你的程式碼能同時跑在 100 台電腦而不出錯，那它就是合格的分散式架構。
- 分散式爬蟲速度極快，如果不受控，瞬間幾千個請求會把別人的伺服器癱瘓（類似 DDoS 攻擊）。可以利用 Redis 的 INCR 指令做頻率限制 (Rate Limiting)。例如：限制同一個 domain 每秒只能抓 2 次，所有 Worker 在抓取前都要先問過 Redis：「我現在可以抓這個網站嗎？」
- 解決「去重爆炸」 (Memory Optimization): 深度技術： 提到你使用了 布隆過濾器 (Bloom Filter)。這是一種機率型資料結構，可以用極小的空間（幾百 MB）判斷一個 URL 是否抓取過，雖然有極低誤判率，但在大規模爬蟲中非常實用。
- 如果某個 Worker 抓到一半電腦炸了，那個網址是不是就丟失了？: 專業回答： 提到 消息隊列的 Ack (確認) 機制。我不會直接把網址從 Redis 刪除，而是移動到一個「處理中」的列表，等 Worker 回報「成功」後才正式刪除。如果沒回報，超時後會由另一個 Worker 接手。
